## 3. AI 답을 그대로 쓰지 않고 검증·거부·수정한 사례 (중요)

- 어떤 답이 틀렸거나 부족했고, 어떻게 알아챘으며, 무엇으로 고쳤나요?
- (이 항목이 비어 있으면 감점 요인입니다. 최소 1개 이상 적어 주세요.)

**1. react-window 버전 불일치로 인한 타입 에러**
AI가 처음 제시한 코드는 react-window v1 API(`FixedSizeList`)를 기준으로 작성됐으나, 실제 설치된 버전은 v2였습니다. `'"react-window"' 모듈에 내보낸 멤버 'FixedSizeList'이(가) 없습니다` 에러로 문제를 발견했고, AI가 웹 검색으로 v2의 새 API(`List`, `rowComponent`, `rowProps`)를 확인한 뒤 코드를 재작성했습니다.

**2. 가상화 카드 높이(itemSize) 임의값 문제**
AI가 카드 높이를 실측 없이 88px, 76px 등 추정값으로 제시했습니다. 실제로는 수정/삭제 버튼이 추가되며 카드 높이가 늘어났는데 값이 갱신되지 않아 카드가 다음 행과 겹치는 문제가 발생했습니다. 브라우저 개발자도구로 직접 렌더링 높이를 실측해 교정했습니다.

**3. 죽은 코드(dead code) — 얼리 리턴과 삼항연산자 중복**
검색 기능 추가 시 AI가 `if (tasks.length === 0) return ...` 얼리 리턴을 그대로 둔 채, 아래쪽 렌더링부에 동일 조건의 삼항연산자를 추가로 제안했습니다. 얼리 리턴에서 이미 해당 케이스가 걸러지므로 아래쪽 분기는 절대 실행되지 않는 죽은 코드였습니다. 코드 흐름을 직접 추적해 문제를 지적했고, 얼리 리턴을 제거하고 렌더링부의 삼항연산자로 통합하도록 수정했습니다.

**4. 이미 제공된 순수 함수를 인지하지 못하고 중복 구현**
AI가 검색 필터링 로직을 신규 함수(`filterTasksByTitle`)로 별도 작성했으나, 이후 `src/lib/tasks.ts`에 동일한 역할의 `filterByTitle`, `moveTask`가 이미 준비되어 있고 테스트까지 작성되어 있었음을 확인했습니다. 기존 파일 내용을 AI에게 직접 공유해 중복 파일을 제거하고, 기존 함수를 재사용하도록 리팩토링했습니다.

**5. 훅 분리 리팩토링 중 useState 선언 누락**
mutation 로직을 커스텀 훅으로 분리하는 과정에서, AI가 기존에 쓰이던 `isFormOpen`, `editingTask` 상태의 `useState` 선언을 누락한 채 코드를 제시했습니다. 실행 시 `is not defined` 에러가 날 것을 예상하고 지적했고, 선언을 추가해 해결했습니다.

## 4. 스스로 판단해서 결정한 부분

- AI 제안과 다르게 결정한 지점과 그 이유.

- **409 충돌 처리 방식 선택**: 리팩토링 과정에서 AI가 이전과 다른(더 단순화된) 버전의 409 처리 코드를 제시했습니다. 두 코드가 다르다는 걸 발견하고 AI에게 차이를 물어봐서 "재조회 없이 payload.current를 즉시 반영하면 네트워크 왕복을 줄일 수 있다"는 설명을 들었고, 이를 검토한 뒤 더 효율적인 방식을 유지하기로 스스로 결정했습니다.
- **검색 디바운싱 미적용**: AI가 필요 여부를 물었을 때, 5,000개 규모에서 필터 연산이 실측상 충분히 빨라 디바운싱 없이도 무리 없다고 판단해 Priority 1에 시간을 더 쓰는 쪽으로 결정했습니다.
- **테스트 대상 선정**: AI가 처음엔 화면(JSX) 레벨의 통합 테스트를 시사했으나, React Query 등 의존성이 많아 시간 대비 효율이 낮다고 판단해 순수 함수(moveTask, filterByTitle, mergeServerTask) 위주로 테스트 범위를 결정했습니다.
