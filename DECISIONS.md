# 설계 결정 (DECISIONS)

> 이 문서는 평가에서 **20%**를 차지합니다. "무엇을 했는지"보다 **"왜 그렇게 했는지"**를 봅니다.
> 아래 항목을 채워 주세요. 코드로 답이 되는 부분은 파일 경로를 함께 적어도 좋습니다.

## 1. 상태 구조

- 서버 상태와 클라이언트(낙관적) 상태를 어떻게 나눴나요? 어디서 관리하나요?
- 이렇게 잡은 이유와, 고려했다가 버린 대안은?

서버 상태는 전부 React Query의 캐시(`queryKey: ['tasks']`)로 관리하고, 별도의 로컬 `useState`로 태스크 목록을 이중 관리하지 않았습니다. 낙관적 변경도 이 캐시를 `queryClient.setQueryData`로 직접 수정하는 방식으로 처리했습니다 (`src/hooks/useTaskMutations.ts`).

- **이유**: `useState`로 관리하면 로딩/에러/재시도/캐시 무효화를 전부 손으로 구현해야 하고, 서버 데이터와 UI 상태가 뒤섞이기 쉽습니다. React Query 캐시 하나만 데이터의 진짜 출처로 두면 서버 상태와 클라이언트 상태의 경계가 명확해집니다.
- **버린 대안**: 초기에는 `Board.tsx` 내부에서 `useState`로 태스크를 관리하던 스타터 코드 구조를 유지할지 고민했으나, 낙관적 업데이트/롤백/경쟁 상태를 직접 구현하면 코드가 복잡해지고 버그 위험이 커 React Query 전환을 택했습니다.

## 2. 낙관적 업데이트 & 롤백

- 이동/수정/삭제/생성 시 UI를 먼저 반영하고 실패하면 어떻게 되돌리나요?
- 롤백을 "서버 재조회" vs "이전 스냅샷 복원" 중 무엇으로 했고 그 이유는?

이동·생성·수정·삭제 4개 동작 모두 `onMutate`에서 캐시를 먼저 낙관적으로 변경하고 `onError`에서 `onMutate`가 반환한 스냅샷(`previousTasks`)으로 정확히 복원합니다 (`src/hooks/useTaskMutations.ts`).

- **롤백 방식**: "서버 재조회"가 아니라 "이전 스냅샷 복원"을 택했습니다. 재조회는 실패 직후 다시 네트워크 요청을 보내야 해서 느리고, 그 사이 화면이 잘못된 상태로 잠깐 노출될 수 있습니다. 스냅샷 복원은 로컬에 이미 갖고 있는 데이터로 즉시 되돌릴 수 있어 더 빠르고 예측 가능합니다.
- **생성(create)의 경우**: 실제 서버 id가 없는 상태이므로 임시 id(`temp-${uuid}`)를 부여한 낙관적 태스크를 캐시에 추가하고, 실패 시 스냅샷으로 롤백, 성공 시 `onSettled`의 재검증으로 실제 서버 데이터(id 포함)로 교체되도록 했습니다.

## 3. 경쟁 상태(race condition)

- 같은 카드를 빠르게 연속 이동할 때 최종 상태가 어긋나지 않도록 무엇을 했나요?
  (요청 취소 / 버전·순번 비교 / last-write-wins 등)

같은 카드를 빠르게 연속 이동시키면 응답이 요청 순서와 다르게 도착할 수 있습니다. 이를 위해 두 단계로 대응했습니다:

1. `onMutate`에서 `queryClient.cancelQueries`로 진행 중인 background refetch를 취소 — 낡은 응답이 방금 반영한 낙관적 상태를 덮어쓰는 것을 방지
2. `onSettled`에서 `invalidateQueries`로 무조건 서버 최신 상태와 재동기화 — 성공/실패와 무관하게 최종적으로는 항상 서버 상태로 수렴

즉 "요청 취소" + "최종 재검증"의 조합으로 처리했으며 별도의 버전/순번 비교 로직을 직접 구현하지는 않았습니다 (React Query의 쿼리 무효화 메커니즘에 위임).

## 4. 대량 데이터 성능 (5,000개)

- 검색/필터/드래그를 부드럽게 만들기 위해 무엇을 선택했고 **무엇을 포기**했나요?
- 가능하면 측정 근거(예: 최적화 전/후 체감·프로파일)를 남겨 주세요.

`react-window`(v2, `List` 컴포넌트)로 가상화를 적용해 화면에 보이는 카드만 실제 DOM에 렌더링하도록 했습니다 (`src/components/Column.tsx`).

- **드래그 앤 드롭과의 호환성**: 이 프로젝트의 `onDrop` 핸들러가 개별 카드가 아닌 컬럼(`<section>`) 전체에 걸려 있어 가상화 시 흔한 문제(화면 밖 카드로 드래그 불가)가 애초에 발생하지 않는 구조임을 코드 분석으로 확인한 뒤 가상화를 적용했습니다.
- **카드 높이**: `Card` 컴포넌트가 `description`을 렌더링하지 않는 단순 구조라, 제목에 CSS로 줄바꿈 제한(`text-overflow: ellipsis`)을 걸어 높이를 고정할 수 있었습니다. 이에 따라 `VariableSizeList` 대신 구현이 간단한 `FixedSizeList`(v2의 `List`) 방식을 택했습니다.
- **검색 성능**: 필터링을 `useMemo`로 감싸 `tasks`나 검색어가 바뀔 때만 재계산하도록 했습니다. 5,000개 규모에서 `Array.filter`는 수 ms 내로 끝나 별도 디바운싱 없이도 체감 지연이 없었습니다.
- **포기한 것**: 디바운싱, 가변 높이 카드 지원(description 표시 등)은 시간 제약과 현재 데이터 구조상 필요성이 낮아 구현하지 않았습니다.
- **측정 근거**: 별도의 성능 프로파일링은 진행하지 않았습니다. 로컬 환경(제 컴퓨터, 최신 사양)에서는 가상화 적용 전에도 5,000개 규모에서 스크롤/드래그가 크게 버벅이지 않아 체감 차이가 뚜렷하지 않았습니다. 다만 저사양 기기나 다른 브라우저 환경에서는 차이가 있을 수 있다고 판단해 요구사항에 명시된 대로 가상화를 적용해 잠재적 성능 저하에 대비했습니다.

## 5. 정답이 없던 결정들

- **네트워크 완전 단절** 시 동작을 어떻게 정의했나요? 왜?
- **409 충돌** 시 사용자에게 무엇을 보여주고 어떻게 해소하나요? 왜?
- 실패한 쓰기의 **재시도**: 자동 vs 수동 — 무엇을 택했고 이유는?

**네트워크 완전 단절**: 별도 감지 로직 없이, 일반 쓰기 실패(15%)와 동일하게 처리됩니다 — 낙관적 반영을 롤백하고 alert로 안내합니다. 오프라인 감지 후 큐잉·자동 재전송은 구현하지 않았습니다. mock API 환경에서 완전 단절을 신뢰성 있게 재현하기 어려웠고, Priority 1의 15% 실패율 대응을 우선했습니다.

**409 충돌**: 서버가 409 응답에 최신 상태(`payload.current`)를 함께 반환하므로, 별도 재조회 없이 그 값을 캐시에 즉시 반영하는 방식을 택했습니다 (`mergeServerTask`, `src/lib/tasks.ts`). 사용자에게는 "다른 곳에서 먼저 변경되었습니다. 최신 상태로 갱신했습니다" alert로 안내합니다. 내가 바꾼 내용과 서버에 이미 반영된 내용을 나란히 비교해서 보여주고 사용자가 직접 고르게 하는 화면은 만들지 않았습니다 — 서버 상태를 그대로 우선시하는 게 이 과제 규모에서는 더 단순하고 예측 가능한 정책이라 판단했습니다.

**재시도 정책**: 읽기(`getTasks`)는 `retry: 2`로 자동 재시도하되, 쓰기(생성/수정/삭제/이동)는 자동 재시도하지 않고 사용자가 직접 다시 시도하도록 했습니다. 목록을 다시 불러오는 건 몇 번을 다시 해도 그냥 같은 데이터를 다시 받아올 뿐이라 여러 번 재시도해도 안전하지만, 쓰기 작업을 자동으로 재시도하면 혹시 첫 요청이 서버에서는 이미 처리됐는데 응답만 못 받은 경우일 때 같은 요청이 중복으로 들어갈 위험이 있다고 판단했습니다.

## 6. 그 외 트레이드오프 / 남은 한계

- 시간 제약으로 포기한 것, 알고 있는 버그·한계, 더 있었다면 했을 것.

- **테스트 범위**: React Query 훅에 강하게 결합된 로직(mutation 콜백 전체)은 통합 테스트 셋업 비용이 커서, 그 안에서 사용하는 핵심 순수 로직(`moveTask`, `filterByTitle`, `mergeServerTask`)만 분리해 유닛 테스트로 검증했습니다. mutation 훅 자체의 통합 테스트(`renderHook` + `QueryClientProvider` 목킹)는 시간 제약으로 진행하지 못했습니다.
- **409 충돌 시 병합 UI 미구현**: 충돌 시 서버 상태로 덮어쓰는 방식만 적용했고, 내가 바꾼 내용과 서버 내용을 나란히 비교해서 사용자가 직접 선택하게 하는 화면은 만들지 않았습니다. 서버 상태를 우선시하는 단순한 정책이 이 과제 규모에서는 충분하다고 판단했습니다.
- **Priority 2 대부분 미구현**: 검색 디바운싱을 제외한 재시도/백오프, 다중 탭 동기화, 키보드 접근성, 다중 필터는 시간 제약상 Priority 1 완성도를 우선하며 구현하지 못했습니다.
- **알고 있는 한계**: 낙관적으로 생성된 태스크(`temp-` id)가 서버 요청 완료 전 사용자가 곧바로 수정/삭제를 시도할 경우의 동시성은 별도로 처리하지 않았습니다. `onSettled`의 재검증 시점에 최종적으로는 정합성이 맞춰지지만, 그 사이 짧은 시간 창에 대한 명시적 가드는 없습니다.
- **더 있었다면 했을 것**: 네트워크 완전 단절 감지 후 자동 재전송, 실제 성능 측정(개발자도구 프로파일링), 다중 탭 동기화, 키보드 접근성 대응
